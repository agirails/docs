`@agirails/sdk@3.3.0` ships real x402 v2 protocol support — the on-the-wire shape Coinbase's facilitator actually expects, not our pre-spec adapter. Plus seller-side server module and Smart Wallet Permit2 batched payment, verified live on Base Sepolia.

## Install

```bash
npm install @agirails/sdk@3.3.0
```

## Three-block release

### Block A — Buyer-side x402 v2 (real protocol)

Earlier `X402Adapter` versions implemented an AGIRAILS-native shape that was *inspired by* x402 but didn't interoperate with the public Coinbase facilitator. Block A rewrites the buyer flow to the actual x402 v2 spec:

```typescript
// HTTPS endpoint? Auto-routed to x402 v2.
await client.pay({
  to: 'https://api.example.com/translate?text=hello',
  amount: '0.10',
});
// 1. GET endpoint → 402 Payment Required + accepts[]
// 2. Buyer constructs PaymentPayload per spec
// 3. POST endpoint with X-PAYMENT header
// 4. Receives X-PAYMENT-RESPONSE with settlement proof
```

Six audit rounds shaped the buyer side: asset allowlist, on-chain allowance precheck, opt-in auto-pay, response validation, content-type assertion, EOA sort, payTo optional handling, getStatus restructure to handle facilitator-default behaviors.

### Block B — Seller-side server module (`@agirails/sdk/server`)

You can now BE an x402 server, not just a buyer. New subpath export:

```typescript
import { createX402Handler } from '@agirails/sdk/server';

const handler = createX402Handler({
  price: { amount: '0.10', currency: 'USDC' },
  payTo: '0xMyAgent...',
  network: 'base-sepolia',
  asset: 'USDC',
  // ... covenant, expected I/O, optional verifier callback
});

// Wire into Express, Next.js, Cloud Function — anywhere
app.post('/translate', handler.middleware, async (req, res) => {
  // req.x402.settled === true here
  res.json({ translation: '...' });
});
```

The handler implements the full x402 server protocol: 402 challenge generation, payment payload verification (signature recovery, asset allowlist, amount check, on-chain allowance simulation, facilitator settlement), and structured response with `X-PAYMENT-RESPONSE`. Charge once, deliver content; reject and walk away on any verify failure.

### Block C — `actp init` x402 seller snippet + CHANGELOG

`actp init` now offers an x402 seller scaffold as one of the four templates (alongside ACTP earner, ACTP buyer, both). The scaffold generates a working Express server with a single paywalled endpoint, ready to deploy to Vercel/Railway/Hetzner.

## Smart Wallet Permit2 — single-tx UX for x402 buyers

Buyers using Smart Wallets (Tier 1 / 2) now batch the **approve + transfer + settle** sequence into a single ERC-4337 userOp via Permit2. Pre-Permit2 you needed:

1. EOA approve USDC spend → 1 tx, gas
2. Buyer signs payment → off-chain
3. Facilitator settles → 1 tx, gas

With Permit2 routed through the Smart Wallet:

1. Single userOp containing approve + payment authorization → 1 tx, sponsored gas
2. Facilitator settles → 1 tx (provider's choice of paymaster)

Live E2E verified on Base Sepolia between two test wallets — total wall-clock 2.4s for full pay-deliver-settle round trip. (See commit `2452c6c` for the test fixture.)

## Security audit rounds 4-6 closed

This release went through three more audit rounds (rounds 4, 5, 6), closing 18 net findings:
- Asset null bypass (round 6)
- getStatus restructure (round 6)
- Facilitator default handling (round 6)
- EOA sort, content-type, getStatus catch, price zero (round 5)
- P1/P2 hardening across buyer, seller, client (round 4)

## What's deprecated

`X402Relay` and `X402FeeBreakdown` are now marked legacy. The original AGIRAILS-side relay was useful when the public facilitator had availability gaps; with Coinbase's mainnet facilitator stable since February, we're consolidating on the public path. Existing X402Relay deployments continue to work — just no new features.
