## Source-of-truth context for auditors

The audit MUST be done against the **new mental model**, not the SDK-as-it-was-when-this-doc-was-written model.

### New paradigm (current as of 2026-05-26)

- **AGIRAILS.md** is a 1242-line canonical protocol spec, not a config file. Lives at `Platform/agirails.app/web/public/protocol/AGIRAILS.md`. Owners do **not** write it from scratch — they tell their AI assistant ("Claude, Cursor, Cline, any LLM with shell access") to onboard them using the spec. The spec has an embedded `onboarding:` YAML block defining the Q&A flow (name, intent, capabilities, price, network, wallet, etc).
- **Onboarding produces two files**:
  1. **Owner AGIRAILS.md** — template with answers filled in, kept locally
  2. **`{slug}.md`** = the "agent.md" / business card / V4 identity file — public, machine-parseable, used for discovery. Generated by `Platform/agirails.app/web/lib/identity-file-generator.ts`. SDK parses with `parseAgirailsMdV4` (`sdk-js/src/config/agirailsmdV4.ts`).
- **Wallet generation is automatic** — `wallet: generate` in the onboarding block creates `.actp/keystore.json` (chmod 600), auto-generates password, stores in `.env`. Owner never sees the password. Alternative: `wallet: existing` via env var.
- **Code-first SDK usage is the ADVANCED path**, not the default. Most users never touch SDK directly — they let the LLM execute `actp publish`, `actp serve`, etc.

### Current shipped surface (what docs MUST reflect)

- `@agirails/sdk@4.0.0` (TypeScript) + `agirails@3.0.1` (Python) on respective registries. Both via OIDC Trusted Publisher.
- **V3 contracts on Base mainnet** (kernel `0x048c811352e8a3fECd5b0Ec4AA2c2b94083CC842`), V4 on Base Sepolia (kernel `0x9d25A874f046185d9237Cd4954C88D2B74B0021b`). 21-field `TransactionView`, AIP-14 dispute bonds, MIN_FEE on-chain, INV-30 locked-bps.
- **Smart Wallet path end-to-end** (`wallet="auto"`): ERC-4337 + Coinbase paymaster, gasless. `pay`, `accept_quote`, `link_escrow`, `transition_state`, `release_escrow` all route through bundler + paymaster. `msg.sender == requester` on-chain.
- **AIP-2.1 quote channel**: `CounterOfferBuilder` / `CounterAcceptBuilder` + `actp serve` FastAPI daemon.
- **Web Receipts**: EIP-712 `ReceiptWrite` signing + upload to `agirails.app/api/v1/receipts`.
- **Hash-based service routing**: `keccak256(service_name)` on-chain, no JSON metadata in the routing key.
- **Five new CLI commands** (since SDK 3.x): `actp serve`, `actp request`, `actp verify`, `actp claim-code`, `actp repair`.
- **x402 on mainnet**: direct buyer → seller via `@x402/fetch` + facilitator, zero AGIRAILS fee. Relay is sepolia-only opt-in.

### Audit row schema

Each audited file must produce a row with:

| Column | Values |
|---|---|
| `path` | relative path from `docs-site/docs/` |
| `status` | CURRENT / STALE / WRONG / MISSING-SECTIONS / OBSOLETE |
| `mental_model` | OLD (SDK-first / pre-V3) / HYBRID / NEW (AGIRAILS.md-first) |
| `versions_referenced` | list of version strings found ("@agirails/sdk@3.3.0", "agirails==2.3.1", etc.) |
| `code_examples_count` | int |
| `broken_examples` | list with line numbers + why broken |
| `missing_concepts` | which of {Smart Wallet, wallet="auto", Web Receipts, AIP-2.1, actp serve/request/verify/claim-code/repair, LLM onboarding, agent.md identity file, V3 addresses, hash-routing, x402 v2} should be there but isn't |
| `severity` | P0 (broken/misleading on copy-paste or contradicts current SDK) / P1 (stale but works) / P2 (cosmetic) |
| `effort` | S (<30 min surgical) / M (1-2 h rewrite) / L (>2 h, conceptual rework) |
| `notes` | freeform — specific quotes, bug-on-page, recommendation |

Mass severity bias: when in doubt, mark **P0 if a fresh integrator could copy-paste broken code or be misled by stale concept**, P1 if it works but reads as outdated, P2 if pure cosmetic.

### After the audit

Three audit files get rolled up into `SUMMARY.md` with:
- P0 count + list
- P1 count + list
- "What concepts are missing from docs entirely" cross-cut
- "What concepts no longer exist in SDK but docs claim" cross-cut
- Effort total + recommended wave breakdown

Then `IA_PROPOSAL.md` proposes the new sidebar tree based on findings.
